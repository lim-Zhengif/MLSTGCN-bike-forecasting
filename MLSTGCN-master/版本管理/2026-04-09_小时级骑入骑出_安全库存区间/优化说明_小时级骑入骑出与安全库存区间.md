# 2026-04-09 小时级骑入骑出与安全库存区间优化说明

## 目标

这个版本不是替换原来的日级 `bike` 模型，而是在 `2026-04-07_优化版_鲁棒损失_图稀疏化_星期嵌入` 基础上，单独扩展一条更适合调度的小时级主流程：

- 预测明天 24 个小时的骑出量和骑入量
- 根据 24 小时净流量曲线推导安全库存区间 `[S_min, S_max]`

## 新增文件

- `hourly_pipeline_utils.py`
  - 小时级流程的公共工具
  - 包含路径解析、订单聚合、日级已知特征合并、图复制、样本切分、安全库存区间计算

- `prepare_bike_hourly_data.py`
  - 从订单流水聚合出小时级标签
  - 生成小时级训练集
  - 输出新的小时级数据目录与图目录

- `prepare_bike_hourly_training_data.py`
  - 小时级数据准备入口包装

- `train_bike_hourly.py`
  - 小时级训练入口
  - 复用现有训练代码，只把默认数据目录、项目名和预测长度切到小时级

- `evaluate_bike_hourly_date_range.py`
  - 对指定日期区间做小时级滚动预测
  - 输出站点-小时预测表、逐日指标表、逐小时指标表

- `generate_bike_safe_inventory_interval_report.py`
  - 根据小时预测直接计算 `[S_min, S_max]`
  - 同时和真实小时曲线推导出的区间进行对比

## 核心改动

### 1. 标签从日级变成小时级

旧版输出：

- `[B, 1, N, 2]`

新版输出：

- `[B, 24, N, 2]`

其中：

- `24` 对应明天 `00:00~23:00`
- `2` 对应 `小时骑出量` 与 `小时骑入量`

### 2. 输入时间窗改成 7 天逐小时序列

默认：

- `hist_len = 24 * 7 = 168`
- `pred_len = 24`

也就是：

- 输入最近 7 天逐小时历史
- 输出明天 24 小时曲线

### 3. 历史输入特征

默认历史特征包括：

- `小时骑出量`
- `小时骑入量`
- `小时净流量`
- `小时`
- `星期几`
- `是否周末`
- `是否节假日`
- 天气特征

### 4. 目标日已知信息继续接入

目标日可知特征默认继续保留：

- `早晨初始车辆数`
- `总桩数`
- `星期几`
- `是否周末`
- `是否节假日`
- 目标日天气

同时沿用上一版的覆盖率过滤机制：

- 当某类已知特征历史覆盖率过低时，自动跳过
- 这样不会把极低覆盖特征强行喂给模型

### 5. 输出继续加非负约束

小时级输出仍然使用 `Softplus`：

- `pred_小时骑出量 >= 0`
- `pred_小时骑入量 >= 0`

这对计数型流量预测是必要的。

### 6. 引入安全库存区间

令：

- `n_h = in_h - out_h`
- `N_t = sum(n_1 ... n_t)`

则：

- `S_min = max_t(-N_t)`：为避免任意时刻缺车，凌晨至少需要多少车
- `S_max = C - max_t(N_t)`：为避免任意时刻爆仓，凌晨最多能放多少车

其中 `C` 为总站容量。

这样，调度不再只看一个日净流量，而是直接拿到：

- 最低安全备车量
- 最高安全放车量

## 默认输出目录

为了不影响原来的日级产物，这一版默认输出到新的独立目录：

- `data/temporal_data/bike_hourly_safe_inventory`
- `data/graph/bike_hourly_safe_inventory`
- `data/SE/se_bike_hourly_safe_inventory.csv`

因此不会覆盖原来的：

- `data/temporal_data/bike`
- `data/graph/bike`
- `data/SE/se_bike.csv`

## 推荐运行顺序

```bash
python prepare_bike_hourly_training_data.py
python train_bike_hourly.py --device auto --epochs 40 --batch_size 8 --logger csv
python evaluate_bike_hourly_date_range.py --device auto --start_date 2026-02-01 --end_date 2026-02-10
python generate_bike_safe_inventory_interval_report.py --eval_dir outputs/bike_hourly_eval_2026-02-01_to_2026-02-10
```

## 适用场景

这一版更适合回答：

- 明天早高峰会不会缺车
- 明天晚高峰会不会爆仓
- 某站点虽然日净流量预测正确，但小时结构是否错了
- 卡车凌晨应该补车还是先拉走一部分车

## 当前限制

- `早晨初始车辆数` 在 2026-02-01 之前覆盖率仍然偏低，默认可能被自动过滤
- 小时级样本更多、时间窗更长，训练时间会明显高于日级版本
- 本版本复用了原来的 5 张图，没有重新单独构建小时级图

## 论文可写的表述方向

可以把这一版概括为：

1. 将共享单车需求预测从日级总量扩展到次日逐小时骑入/骑出预测
2. 在多图时空图神经网络输出的基础上，构建面向调度的安全库存区间
3. 将预测结果通过累积净流量曲线转化为可执行的库存调度约束
