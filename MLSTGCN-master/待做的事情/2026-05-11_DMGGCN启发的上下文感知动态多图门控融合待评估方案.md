## 3. 为什么核心卖点可以升级

当前模型已经有五图融合：

```text
dist
neigh
func
bike_heuristic
tempp
```

但目前多图融合主要依赖训练得到的全局参数或全局注意力，尚未充分体现：

```text
不同站点
不同小时
不同星期
不同天气
不同节假日
不同库存/需求状态
```

对图关系重要性的影响。

共享单车场景中，这种影响很明显：

```text
早高峰：OD 通勤图、需求相关图可能更重要。
晚高峰：流向关系可能和早高峰不同。
周末：POI 功能图、休闲区域相似性可能更重要。
雨天：历史时间相似图可能下降，外部环境特征权重上升。
库存紧张：库存状态特征应被更强关注。
```

因此，后续卖点可以不再只是“多图注意力”，而是强调：

```text
模型根据站点、时间、环境、库存与需求状态，动态调整图关系和预测特征来源的重要性。
```

## 4. P1：站点-时间-环境上下文嵌入

P1 是基础模块，用于生成上下文表示。

可包含：

```text
Station Embedding
Time-of-day Embedding
Day-of-week Embedding
Weather/Calendar Embedding
Inventory/Demand State Embedding
```

### 4.1 Station Embedding

用于表示站点自身属性。

可来源于：

- 可学习 station id embedding。
- 站点容量 capacity。
- 经纬度/空间位置。
- POI 或功能区信息。
- 历史活跃度。
- 站点平均骑入、骑出、净流量。

作用：

```text
区分住宅区、办公区、交通枢纽、低频站点、高频站点等不同站点类型。
```

### 4.2 Time-of-day Embedding

用于表示一天中不同小时。

可用：

```text
hour = 0..23
```

或者划分为：

```text
凌晨
早高峰
午间
晚高峰
夜间
```

作用：

```text
刻画早晚高峰和低谷时段的不同出行规律。
```

### 4.3 Day-of-week Embedding

用于表示一周周期。

可用：

```text
Monday ... Sunday
```

也可加入：

```text
是否周末
是否工作日
```

作用：

```text
区分工作日通勤需求和周末休闲需求。
```

### 4.4 Weather/Calendar Embedding

用于表示外部环境。

可包含：

- 温度。
- 降水。
- 降雪。
- 风速。
- 是否节假日。
- 是否周末。

作用：

```text
刻画天气和日历因素对骑行需求、站点相似性和库存风险的影响。
```

### 4.5 Inventory/Demand State Embedding

用于表示当前运营状态。

可包含：

- 最近几小时骑出量。
- 最近几小时骑入量。
- 最近几小时净流量。
- 当前库存或估计库存。
- 库存安全区间位置。
- 是否接近缺车/满桩风险。

作用：

```text
让模型感知当前站点是否处于紧张、空闲、异常波动等状态。
```

## 5. P2：上下文感知动态多图融合

P2 是推荐的主创新模块。

核心思想：

```text
用 P1 得到的 context 生成五张图的动态权重。
```

设五张图为：

```text
A_dist
A_neigh
A_func
A_heuristic
A_tempp
```

上下文向量为：

```text
c = ContextEncoder(station, time, day, weather, inventory, demand)
```

通过 MLP 或门控网络生成权重：

```text
alpha = softmax(MLP(c))
```

得到：

```text
alpha_dist
alpha_neigh
alpha_func
alpha_heuristic
alpha_tempp
```

最终动态图：

```text
A_context =
    alpha_dist      * A_dist
  + alpha_neigh     * A_neigh
  + alpha_func      * A_func
  + alpha_heuristic * A_heuristic
  + alpha_tempp     * A_tempp
```

可解释性：

```text
早高峰时，模型可能提高 OD 图或需求相关图权重。
周末时，模型可能提高功能图或 POI 相关图权重。
雨天时，模型可能降低历史相似图权重，提高外部环境相关特征影响。
```

### 5.1 推荐先做的轻量版本

第一版不建议直接为每个样本、每个节点生成完整的 N x N 动态图。

推荐先做：

```text
样本级动态五图权重
```

即每个 batch/sample 生成一组五图权重。

优点：

- 改动相对较小。
- 保持五图结构不变。
- 便于做消融。
- 便于解释不同时间/天气下图权重变化。

后续可扩展为：

```text
节点级动态五图权重
```

即不同站点也拥有不同的图偏好。

## 6. P3：DDGA 风格特征门控融合

P3 是推荐的增强模块。

它和 P2 不冲突：

```text
P2 负责融合图关系。
P3 负责融合预测特征来源。
```

共享单车预测中，输入信息来源很多：

- 历史骑入/骑出。
- 当前或估计库存。
- 时间周期。
- 天气。
- 节假日。
- 站点容量。
- POI/功能区。
- OD 转移。
- 邻近站点状态。

如果只是简单拼接，解释性不足。

可以参考 DDGA 思想，将特征拆成若干来源：

```text
h_s：空间图特征
h_t：时间序列特征
h_i：库存/供需状态特征
h_e：外部环境特征
```

通过门控网络生成：

```text
g_s, g_t, g_i, g_e = softmax(MLP(context))
```

融合：

```text
h_fused =
    g_s * h_s
  + g_t * h_t
  + g_i * h_i
  + g_e * h_e
```

解释性：

```text
通勤时段：空间 OD 与时间周期特征权重更高。
雨雪天气：外部环境特征权重更高。
库存紧张站点：库存状态特征权重更高。
常规平稳时段：历史时间特征权重更高。
```

## 7. P4：horizon-aware 趋势对齐预测头

P4 是可选增强，不建议作为第一主线。

原因：

当前模型已经是并行输出未来多个小时：

```text
output shape = [batch, pred_len, num_nodes, out_dim]
```

因此当前模型不是自回归式预测，不存在严重的逐步输入误差累积问题。

但仍可借鉴 DMGGCN 的趋势对齐思想：

```text
不同预测步 h=1..6 应关注不同历史趋势和未来时间条件。
```

可设计：

```text
horizon embedding
future time embedding
horizon-specific decoder
```

示意：

```text
hidden_history + horizon_embedding[h] + future_time_embedding[h]
    -> decoder_h
    -> y_hat_h
```

作用：

```text
改善 h1 到 h6 中远期预测误差逐步增大的问题。
```

该模块可在 P1+P2+P3 之后再考虑。

## 8. P5：完整 meta-parameter graph convolution

P5 暂缓。

DMGGCN 更强的做法是动态生成或调整图卷积参数，使模型参数也随输入状态变化。

但对当前项目而言，直接实现完整 meta-parameter graph convolution 风险较高：

- 需要较大幅度改动 ChebGCN/MSTGCN。
- 参数规模和显存压力可能上升。
- 训练稳定性不确定。
- 与当前多图融合模块耦合较深。
- 消融和论文解释复杂度较高。

因此不建议作为第一阶段主线。

可作为远期扩展：

```text
如果 P1+P2+P3 已经有效，再考虑局部参数调制，例如 FiLM 风格的 gamma/beta 条件调制。
```

## 9. 推荐落地路线

最优主线：

```text
P1：站点-时间-环境上下文嵌入
    -> P2：上下文感知动态多图融合
    -> P3：DDGA 风格特征门控融合
```

可选补充：

```text
P4：horizon-aware 趋势对齐预测头
```

暂缓：

```text
P5：完整 meta-parameter graph convolution
```

推荐最小实现：

```text
1. 保留原五图结构。
2. 新增 ContextEncoder。
3. 用 ContextEncoder 输出生成五图动态权重。
4. 替代或增强当前 FusionGraphModel 中的全局图权重。
5. 在 MSTGCN 主干后增加 DDGA 风格门控融合。
6. 先不改 ChebGCN 参数生成方式。
```

## 10. 推荐实验设计

建议按消融逐步验证：

```text
Baseline：当前模型
P1 only：只加入上下文嵌入，不动态融合
P1 + P2：上下文感知动态五图融合
P1 + P2 + P3：再加入 DDGA 风格特征门控
P1 + P2 + P3 + P4：再加入 horizon-aware decoder
```

不建议直接上：

```text
P1 + P2 + P3 + P4 + P5
```

因为无法判断收益来自哪里，且工程风险过高。

指标：

- overall MAE/RMSE。
- ride_out MAE。
- ride_in MAE。
- horizon 1-6 MAE。
- anchor hour MAE。
- peak window MAE。
- station-level MAE。
- 调度仿真中的缺车/满桩风险变化。

可解释性分析：

- 不同 anchor hour 下五图权重变化。
- 工作日 vs 周末的五图权重变化。
- 雨天 vs 非雨天的特征门控变化。
- 库存紧张站点 vs 普通站点的库存特征权重变化。

## 12. 暂不实施原因

当前阶段先记录，不立即实现，原因如下：

- P1+P2+P3 涉及模型结构改造，需要谨慎设计接口。
- 当前模型已有较多实验结果，直接大改可能影响已有主线可比性。
- P3 需要拆分特征来源，当前 MSTGCN 主干并非天然多分支结构。
- P4 虽有价值，但当前模型已是并行多步输出，不是最紧急短板。
- P5 工程风险较大，暂不适合作为近期目标。

## 13. 后续优先级

推荐后续顺序：

```text
1. 先画出 P1+P2+P3 的模块结构图。
2. 明确哪些已有输入特征可用于 context encoder。
3. 先实现样本级动态五图权重，而不是节点级完整动态图。
4. 做 P1+P2 的小规模实验。
5. 若 P1+P2 有收益，再加入 P3。
6. 若 h6 等远期误差仍明显，再考虑 P4。
7. P5 只作为远期高风险方案保留。
```

最保守第一步：

```text
只做 ContextEncoder + 动态五图权重。
```

这一步最贴近当前模型，也最容易形成清晰消融。
